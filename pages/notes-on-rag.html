Chapter VI · 第一卷 · 第六章 · 2026.03 · ~10 min read

架构视角

从知识的版本、引用、按需加载,看 RAG 改变了什么、没改变什么

Architecture RAG Memory

I.RAG 解决了什么

一个朴素的问题:模型"知道"的事情是有限的(它的训练数据截止那一刻为止), 而且它在那些事上的记忆也是压缩的、扭曲的(参数量是有限的)。

RAG(Retrieval-Augmented Generation)的回应很直接: 别让模型什么都记,让它在需要的时候去查。 它把"知识"从模型参数里搬到了外部存储里。

  • 想引用最新论文?查 arXiv;
  • 想回答公司内部问题?查 Confluence;
  • 想引用某段具体文档?查向量数据库。

模型的注意力被"按需支付"在真正相关的知识上 —— 而不是把所有可能性塞进上下文窗口。

II.知识的三种状态

从架构视角看,RAG 让知识有了三种状态:

  1. 参数化的记忆(在模型权重里):快、模糊、难改;
  2. 上下文窗口(在当前 prompt 里):精确、有上限、易失;
  3. 外部知识库(在 RAG 后端):无限、可改、可治理。

这是范式变化。原来只有前两种;现在有了第三种,而且第三种会越来越主导。

III.引用与可追溯性

RAG 最大的隐性收益,是引用。 以前模型"凭印象"回答,出错你无据可查。 现在模型给答案,顺手给几个来源,你点过去就是原文。

可追溯,等于可证伪。可证伪,等于可信任。

但这是有代价的:你的知识库必须可治理—— 来源要可信,版本要可追,访问要可控。 一份被污染的知识库,会让模型系统性地胡说八道。

IV.按需加载的工程

RAG 在工程上是"按需加载"的极致实现: 

用户问 → 改写为检索 query → 向量检索 → 重排
→ 取出 top-k 文档 → 拼进 prompt → 模型生成

每一步都是一个独立的优化空间: 改写、embedding、chunking、reranking、压缩、citation。 每一步都有自己的坑,有自己的论文。

V.RAG 没改变什么

实话:RAG 没改变模型本身的智能上限。 它改变的是:在特定任务上的表现、对新鲜事物的响应速度、可治理性。

一个模型该胡说的地方,RAG 也救不了; 一个本来就不会的推理,RAG 也教不会。 RAG 是脚手架,不是地基。

VI.一个判断

我赌:未来三年,会形成一种"模型 + RAG + 工作流"的三层架构:

  • 模型:负责通用智能;
  • RAG:负责事实与新鲜度;
  • 工作流(Agent / Skill):负责稳定可重复的过程。

把这三层当一个 stack 来设计、来运维、来治理, 会变成 Agent 时代最基本的工程素养。
