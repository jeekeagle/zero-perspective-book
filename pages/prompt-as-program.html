程序视角

把提示词当作源码,看它如何获得类型、模块、版本这些程序性特征

Programming Prompt DSL

I.从自然语言到 DSL

早期写提示词,和写一段"恳切的信"差不多:礼貌、详细、希望对方能懂。 这种提示词就像一份没有注释、没有类型、全部塞在一个文件里的 PHP 早期代码—— 能跑,但没人能维护。

今天的情况变了。一个严肃的 Prompt 工程,正在向一种新的领域特定语言(DSL)演化: 它有结构(章节分明的 System / Developer / User)、有变量(模板插值)、有函数(可调用的子 Prompt), 有类型(结构化输出)、有测试(eval suite),甚至有类型检查器(JSON schema + validator)。

II.提示词作为源码

把一段提示词拆开看,它已经有这些程序特征:

  • 类型:JSON schema、enum、format;
  • 函数:@tool、@skill、sub-prompt;
  • 模块:import 别人的 prompt 或 tool;
  • 版本:Git 仓库里的 v1.2.3;
  • 测试:eval suite、golden set;
  • 运行时:Prompt → LLM → Tool → 反馈环。

你几乎可以把它当作一种"运行在大模型上的脚本语言"。

III.版本管理

这是被严重低估的一环。 一段 prompt 的"v1"和"v2"在行为上可能差异巨大(模型换了、上下文窗口变了、加了示例)。 但很多人还在用文件名后缀(prompt-v3-final.md)管它。

没有版本管理的 prompt,迟早会让团队"以谁的版本为准"开一场永远开不完的会。

IV.提示词的"运行时"

一段 prompt 不是孤立执行的,它有自己的运行时: 

用户输入 → 模板渲染 → 上下文组装 → 模型调用
                                ↓
                          工具调用 / 子 prompt
                                ↓
                          结果解析 + 校验
                                ↓
                            输出 / 失败重试

看清这条链路,你就理解了为什么"调 prompt"常常不是改文本, 而是在改这条链路里的某一段(改模板、改上下文、改工具、改错误处理)。

V.它不是程序

我必须诚实地说:prompt 像程序,但不是程序。

  • 没有严格的执行顺序(模型可能"跳过"或"重排");
  • 没有形式化的语义(同一句话可能产生不同的内部解释);
  • 没有确定的输出(同一段 prompt,同样的输入,可能输出不一样)。

它是一种"近似确定性"的程序。这意味着传统的 linter、type checker、formal verification 都失效了,需要新的工具:基于 eval 的回归测试、基于人类偏好的 A/B、基于统计的 CI。

VI.一个判断

我赌:未来三年,会出现"Prompt Engineer ≠ 文案"的认知反转。 真正值钱的 prompt engineer,会越来越像系统程序员 + 实验科学家: 懂类型、懂版本、懂实验设计、懂统计。 而单纯的"调语气、加例子",会被模型自己内化掉。
