拓扑学视角

把 Agent 协作画成图,看串行、并行、投票、监督这些结构背后的组织哲学

拓扑学 结构 Multi-Agent

I.把协作画成图

假设你面前有 N 个 agent,每个 agent 会调一些工具、读一些上下文、产出一些中间结果。 你想理解它们怎么合作 —— 别看代码,看图。

节点是 agent 或工具,边是"调用 / 读 / 写"。 几乎所有 multi-agent 系统都可以归到下面几种基本拓扑上:

  • Pipeline:线性串行,A → B → C;
  • Fan-out / Fan-in:一拖多,合并结果;
  • Hub-and-spoke:一个调度,周围一圈 worker;
  • Mesh:谁都能调谁(也是最容易出乱子的);
  • Loop:自己审自己、自己改自己。

II.拓扑决定性质

拓扑不是装饰。它决定系统的性质。

同样的 agent 能力,放在不同的拓扑里,会做出完全不同的系统。

举三个例子:

  • Pipeline:可控、稳定、易调试,但串行延迟 = 各环节延迟之和;
  • Fan-out:快(并行)、可对比(多视角),但成本 = N 倍;
  • Loop(反思):质量高(自我修正),但不可预测完成时间,还可能死循环。

III.监督关系的拓扑

当 agent 能改 agent 的输出,系统就有了监督关系。 监督关系画出来是树:每个节点最多一个父,但可以有多个子。

这个看似简单的图,藏着组织学的老问题:

  • 层级数:深监督树更稳但更慢;
  • 层级数 = 0(mesh):快但乱;
  • 监督者是否同质:同质监督 → 容易被群体偏见带偏;异质监督 → 慢但稳。

在人类社会里,这些早就被管理学研究透了。 换到 agent 系统里,问题一模一样,只是变量变了(人变成了 LLM 调用,会议变成了 token 流)。

IV.投票:民主的拓扑

让 N 个 agent 独立出方案,再让一个 aggregator 选/合并,这是 multi-agent 系统的"民主化"操作。 它之所以有效,是因为:

  • 去相关性:不同 prompt / 不同样本 → 不同错误模式;
  • 覆盖性:多个视角合起来比单一视角更全;
  • 自我校准:多数派往往比极端派更接近真相。

它的代价:贵(N 倍 token)、慢(N 倍 wall-clock)、可能"多数派暴政"。

V.什么时候不该复杂化

我见过太多"为了 multi-agent 而 multi-agent"的系统。 一个 prompt 能干的事,非要拆 5 个 agent 来做,延迟涨 4 倍、token 涨 6 倍,质量涨 5%。

拓扑的复杂度,不应该超过任务的本质复杂度。 一句话能讲清楚的事,就别开董事会。

VI.一个判断

我赌:2026-2027 年会出现一波"拓扑学复兴"。 大家会发现,multi-agent 的真正难点不是 prompt,而是图。 会有人写"Agent Design Patterns"(对应 GoF 之于 OO), 会有人做"Agent 编排可视化",会有人把组织行为学搬进 agent 系统。

看到这一章的读者,提前补一补图论和分布式系统,会划算。
